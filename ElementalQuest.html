About the game

Elemental Quest is a 3D action-adventure game where you play as Nerlim, a young mage mastering the elements of Fire, Grass, and Water. Princess Lyra has been captured you must journey through the ruined castle of Lord Eclipse, a fallen sorcerer who once tried to fuse Light and Dark magic. Explore overgrown courtyards, flooded docks, and molten forges, using elemental powers to defeat monsters and solve environmental puzzles.

The enemy AI is built across three classes with clear responsibilities. EQEnemy is the character body — it owns the stats, GAS component, and reacts to gameplay events like being hit, stunned, or knocked back. EnemyAIController is the brain — it runs perception, sends events to the StateTree, and coordinates with the BattleCircle system to manage how many enemies attack at once. SkeletonAttackTask is a single StateTree node — it fires the attack ability via GAS and manages the attack cooldown loop.

Enemy detecting, chasing, and attacking the player

EQEnemy.cpp — The Character

EQEnemy.cppC++
// On possession, pass patrol/chase settings to the controller
// then start the StateTree
if (AEnemyAIController* EnemyAI = Cast<AEnemyAIController>(WeakController.Get()))
{
    EnemyAI->SetPatrolSpeed(WeakThis->PatrolSpeed);
    EnemyAI->SetChaseSpeed(WeakThis->ChaseSpeed);

    if (EnemyAI->StateTreeComponent && WeakThis->StateTree)
    {
        EnemyAI->StateTreeComponent->SetStateTree(WeakThis->StateTree);
        EnemyAI->StateTreeComponent->StartLogic();
    }
}

// When hit, check element tag and react — e.g. Grass hit applies a stun
if (AbilitySystemComponent->HasMatchingGameplayTag(Tags.Damage_Grass))
{
    if (!bShieldActive) ApplyStun();
}

What this does

EQEnemy owns the AbilitySystemComponent and AttributeSet — the two core GAS objects. On possession it pushes its designer-set speeds and StateTree asset to the controller, then starts the AI logic.

It also listens to GAS attribute and tag events directly. When health changes, it checks which element tag is on the ASC — Fire, Water, or Grass — and reacts accordingly: breaking a matching shield, applying a stun, or triggering a hit flash. The enemy doesn't know who hit it, it just reads its own tags.

EnemyAIController.cpp — The Brain

EnemyAIController.cppC++
// Perception sees the player — send event to StateTree
if (Stimulus.WasSuccessfullySensed())
{
    CurrentTarget = Actor;
    SetFocus(Actor);

    if (UBattleCircleSubsystem* BCS = GetWorld()->GetSubsystem<UBattleCircleSubsystem>())
        BCS->RegisterEnemy(this);

        FGameplayTag PlayerDetectedTag = FGameplayTag::RequestGameplayTag("AIEventPlayerInSight");
        SendStateTreeEvent(PlayerDetectedTag);
}

// Forward the event into the running StateTree
void AEnemyAIController::SendStateTreeEvent(FGameplayTag EventTag)
{
    if (StateTreeComponent && StateTreeComponent->IsRunning())
    {
        FStateTreeEvent Event;
        Event.Tag = EventTag;
        StateTreeComponent->SendStateTreeEvent(Event);
    }
}

What this does

The controller owns the AIPerceptionComponent and watches for sight stimuli. When the player enters view it sets them as focus, registers with the BattleCircleSubsystem (which limits how many enemies attack at once), and fires the AIEventPlayerInSight tag into the StateTree to trigger the Chase state.

When the player is lost it releases its attack token, clears focus, and sends AIEventPlayerNotInSight — sending the enemy back to patrol. The StateTree never polls the world directly; it only reacts to events the controller pushes to it.

SkeletonAttackTask.cpp — The Action

SkeletonAttackTask.cppC++
// EnterState — bail immediately if enemy is stunned
if (Enemy->IsStunned()) return EStateTreeRunStatus::Failed;

// Tick — fire the GAS attack ability once after a short delay
if (!InstanceData.bHasAttacked && InstanceData.TimeElapsed >= 0.2f)
{
    InstanceData.bHasAttacked = true;
    FGameplayTag AttackTag = FEQGameplayTags::Get().Abilities_Attack;
    ASC->TryActivateAbilitiesByTag(FGameplayTagContainer(AttackTag));
}

// After cooldown — stay in Attack if still in range, exit if not
if (InstanceData.TimeElapsed >= InstanceData.AttackCoolDown)
{
    return (DistanceToTarget <= InstanceData.AttackRange)
    ? EStateTreeRunStatus::Running   // loop attack
    : EStateTreeRunStatus::Succeeded; // back to Chase
}

What this does

This StateTree Task runs while the enemy is in the Attack state. On entry it immediately checks if the enemy is stunned and fails out if so, letting the StateTree transition elsewhere.

Each tick it accumulates time. After 0.2 seconds it fires the Abilities_Attack tag into the enemy's ASC — GAS handles finding and playing the right attack ability. After the full cooldown it checks distance: still close enough means it resets and attacks again; too far means it returns Succeeded, which signals the StateTree to go back to Chase. On exit it releases the BattleCircle attack token so another enemy can take its turn.

How they connect

EQEnemy spawns with its StateTree asset and speeds set in Blueprint → on possession it hands them to EnemyAIController and starts the StateTree → the controller's Perception detects the player and sends AIEventPlayerInSight into the StateTree → StateTree transitions to Chase, then to Attack when in range → SkeletonAttackTask runs and calls TryActivateAbilitiesByTag on EQEnemy's ASC → GAS fires the attack ability → on death, EQEnemy calls NotifyDeath() on the controller which releases the BattleCircle token and sends AIEventDead to cleanly shut the StateTree down.

The Puzzle Manager is a designer-facing actor you drop into any level and configure entirely from Blueprint. You assign whichever puzzle types you need — door switches, grass switches, time puzzles, or sequence puzzles — and the manager auto-registers itself with all of them on BeginPlay. When every assigned puzzle reports as activated, the manager fires its completion event. AppearingActor is what the manager rewards the player with — an actor that dissolves into existence using a dynamic material, complete with sound and Niagara effects.

PuzzleManager.cpp — The Coordinator

PuzzleManager.cppC++
// BeginPlay — auto-registers this manager with every assigned puzzle
for (ADoorSwitch* Switch : RequiredSwitches)
    if (Switch && !Switch->PuzzleManager) Switch->PuzzleManager = this;

for (AGrassSwitch* GrassSwitch : RequiredGrassSwitches)
    if (GrassSwitch && !GrassSwitch->PuzzleManager) GrassSwitch->PuzzleManager = this;

for (ATimePuzzle* TimePuzzleObj : RequiredTimePuzzleObjects)
    if (TimePuzzleObj && !TimePuzzleObj->PuzzleManager) TimePuzzleObj->PuzzleManager = this;

for (ASequencePuzzle* SeqPuzzle : RequiredSequencePuzzleObjects)
    if (SeqPuzzle && !SeqPuzzle->PuzzleManager) SeqPuzzle->PuzzleManager = this;

// CheckPuzzleCompletion — called whenever any puzzle reports a state change
if (bAllActivated && !bPuzzleComplete)
{
    bPuzzleComplete = true;
    ExecutePuzzleComplete();
}
else if (!bAllActivated && bPuzzleComplete)
{
    bPuzzleComplete = false;
    UpdateActorsState(false); // reverse the effect if puzzle broken
}

What this does

On BeginPlay the manager loops through every array the designer filled in — door switches, grass switches, time puzzles, sequence puzzles — and injects itself as their manager reference. No manual wiring per puzzle object is needed.

Every time any puzzle piece changes state it calls CheckPuzzleCompletion, which tallies all arrays in one pass. If everything is activated it fires ExecutePuzzleComplete. Crucially, if a puzzle can be un-solved (a switch toggled off, a plate stepped off), the manager catches that too and reverses the completion effect automatically.

PuzzleManager.cpp — Sequence & Timer Logic

PuzzleManager.cppC++
// OnSequencePuzzleHit — enforce correct order
void APuzzleManager::OnSequencePuzzleHit(ASequencePuzzle* Object)
{
    // Count already-activated objects to know expected index
    int32 ActivatedCount = 0;
    for (ASequencePuzzle* SeqPuzzle : RequiredSequencePuzzleObjects)
        if (SeqPuzzle && SeqPuzzle->IsActivated() && SeqPuzzle != Object)
            ActivatedCount++;

    // Wrong order — reset every object immediately
    if (Object->SequenceIndex != ActivatedCount)
    {
        for (ASequencePuzzle* SeqPuzzle : RequiredSequencePuzzleObjects)
            if (SeqPuzzle) SeqPuzzle->Reset();
            return;
    }

    // First correct hit — start the sequence timer
    if (bUseSequenceTimer)
    {
        GetWorldTimerManager().SetTimer(
            SequenceTimerHandle, this,
            &APuzzleManager::OnSequenceTimerExpired,
            SequenceTimeLimit, false
        );
    }
    CheckPuzzleCompletion();
}

What this does

For sequence puzzles the manager tracks order by counting how many objects are already activated before the new hit arrives. If the incoming object's SequenceIndex doesn't match that count, the order is wrong and everything resets immediately.

If a sequence timer is enabled, the first correct hit starts a countdown. Hit the remaining objects before it expires and the puzzle completes — miss the deadline and OnSequenceTimerExpired resets the whole thing. This all runs in the manager with no extra code needed in the individual puzzle objects.

AppearingActor.cpp — The Reward

AppearingActor.cppC++
// SetupDissolveMaterials — swap every mesh slot to a dynamic dissolve material
for (int32 i = 0; i < ActorMesh->GetNumMaterials(); i++)
{
    UMaterialInstanceDynamic* DynMat = UMaterialInstanceDynamic::Create(DissolveMaterial, this);
    ActorMesh->SetMaterial(i, DynMat);
    DynamicMaterials.Add(DynMat);
}

// UpdateDissolve — ticks at ~60fps, drives the dissolve scalar parameter
CurrentDissolveValue += Direction * DissolveSpeed * DeltaTime;
for (UMaterialInstanceDynamic* DynMat : DynamicMaterials)
    DynMat->SetScalarParameterValue(DissolveParameterName, CurrentDissolveValue);

// OnAnimationComplete — restore originals and enable collision
if (FMath::IsNearlyEqual(CurrentDissolveValue, DissolveEndValue, 0.1f))
{
    bIsVisible = true;
    UpdateCollision(true);
    RestoreOriginalMaterials(); // swap dissolve mat back to original
}

What this does

When the Puzzle Manager calls TriggerAppear(), the AppearingActor stores all original mesh materials, then replaces every slot with a dynamic instance of the dissolve material. A timer running at ~60fps increments a scalar parameter on that material from the start value (invisible) to the end value (fully solid), creating the dissolve-in effect.

Once the animation completes it swaps the original materials back in and re-enables collision — so the appeared object looks and behaves exactly as the artist intended, with no dissolve material left on it. TriggerDisappear() runs the same process in reverse. The designer sets the dissolve speed, emissive glow colour, sound, and Niagara effect all from Blueprint.

How they connect

Designer places a PuzzleManager and assigns any mix of puzzle types in the Inspector → BeginPlay auto-registers the manager with every assigned puzzle → player solves puzzles in the level; each one calls back into the manager → CheckPuzzleCompletion runs after every change and tallies all arrays → when all pass, ExecutePuzzleComplete fires OnPuzzleCompleted and calls TriggerAppear() on any assigned AppearingActors → AppearingActor swaps in the dissolve material, animates the scalar parameter each frame, then restores the original materials once fully visible.

The Universal Trigger is a single, highly flexible actor that handles almost every event-based interaction in the game — enemy wave spawning, combat puzzles, checkpoints, appearing/disappearing actors, and on-screen instructions.

Instead of creating many different trigger classes, I built one powerful trigger that designers can configure entirely in Blueprint. This dramatically reduced blueprint spaghetti and made level design much faster and more consistent.

Key Advantages

• One actor type instead of many → easier maintenance
• Fully configurable in Blueprint (no C++ changes needed for new levels)
• Built-in support for checkpoints, instructions, and AppearingActors
• Clean separation: Trigger → Subsystems (Spawn / Combat Puzzle)
• Supports both one-time and repeatable triggers
• Debug-friendly (P key resets the trigger)

The CombatPuzzleSystem is a powerful, designer-friendly component that turns any arena into a multi-wave combat puzzle. It handles enemy wave spawning, arena wall management, spider/frog baby registration, and seamless progression between waves.

Integrated directly into the Universal Trigger, designers can set up complex combat encounters entirely in Blueprint by configuring waves, spawn points, delays, enemy types, and shields.

// Called by UniversalTrigger when player enters arena

{
    if (bCombatActive || bPuzzleCompleted) return;

What this does

When the player enters the trigger, the system activates the combat arena by raising walls and immediately starts the first enemy wave. It prevents re-triggering once combat begins or the puzzle is solved.

Wave Completion Logic (The Tricky Part)

CombatPuzzleSystem.cppC++
// Called by enemies when they die
void UCombatPuzzleSystem::OnEnemyDied()
{
    EnemiesDied++;
    GetWorld()->GetTimerManager().SetTimerForNextTick([this]() {
    CheckWaveComplete();
    });
}

void UCombatPuzzleSystem::CheckWaveComplete()
{
    if (!bAllEnemiesQueued || bWaveCompletionPending) return;
    if (PendingChildrenToRegister > 0) return;   // Wait for spider babies
    if (EnemiesDied < EnemiesExpectedThisWave) return;

    bWaveCompletionPending = true;
    AdvanceOrCompleteWave();
}

What this does

This is the most sophisticated part. The system intelligently waits until:

  • All enemies have been queued for spawning
  • All baby enemies (spiderlings, etc.) have registered themselves
  • The exact expected number of enemies have died

Using SetTimerForNextTick prevents race conditions when multiple enemies die in the same frame.

Special Features

• Multi-wave support with configurable delay between waves
• Dynamic enemy types (Flower, Spider, Frog, Crab, Golem, FireFrog)
• Shield system — enemies can spawn with Fire/Grass/Water shields
• Baby enemy support — spiders can spawn children that properly register
• Arena walls — automatically raises/lowers walls to trap player during combat
• Reset support — fully resettable for testing or replayability
• Patrol spline integration for enemies that need to follow paths

How it connects to Universal Trigger

When the final wave is cleared, AdvanceOrCompleteWave() calls OnObjectiveComplete() on the owning UniversalTrigger, which then hides instructions and can make reward actors appear.

The checkpoint system allows players to respawn at the last activated checkpoint after death, instead of restarting from the beginning of the level. It is seamlessly integrated into the UniversalTrigger actor, making it extremely easy for designers to place checkpoints throughout the level.

Player dies and respawn at Checkpoint

{

void AUniversalTrigger::ActivateCheckpoint(AEQCharacter* Character)
{
    if (!CheckpointSpawnPoint) return;

    if (APlayerController* PC = Cast<APlayerController>(Character->GetController()))
    {
        if (AEQPlayerState* PS = PC->GetPlayerState<AEQPlayerState>())
        {
            FVector SpawnLoc = CheckpointSpawnPoint->GetActorLocation();
            FRotator SpawnRot = FRotator(0.f, 45.f, 0.f); // Force consistent yaw
            
            PS->SetCheckpointLocation(SpawnLoc, SpawnRot);
        }
    }
}

What this does

When the player walks into a trigger marked as a checkpoint, it saves the location and rotation to the PlayerState. The checkpoint only activates once (unless reset), and the spawn point is taken from a separate actor reference placed by the designer.

GameMode Integration — Respawning at Checkpoint

EQGameModeBase.cppC++
// Override to support checkpoint respawn
AActor* AEQGameModeBase::FindPlayerStart_Implementation(AController* Player, const FString& IncomingName)
{
    if (AEQPlayerState* PS = Player->GetPlayerState<AEQPlayerState>())
    {
        if (PS->HasCheckpoint())
        {
            FVector Loc = PS->GetCheckpointLocation();
            FRotator Rot = PS->GetCheckpointRotation();

            // Spawn a temporary PlayerStart at the checkpoint
            APlayerStart* TempStart = GetWorld()->SpawnActor<APlayerStart>(
            APlayerStart::StaticClass(), Loc, Rot);

            if (TempStart) return TempStart;
        }
    }
    return Super::FindPlayerStart_Implementation(Player, IncomingName);
}

What this does

Before spawning the player (on death or level restart), the GameMode checks if the PlayerState has an active checkpoint. If yes, it dynamically creates a temporary APlayerStart at that exact location and rotation. This gives full control over where and how the player respawns without needing multiple permanent PlayerStarts.

Resetting Combat Puzzles on Death

EQGameModeBase.cppC++
void AEQGameModeBase::RestartPlayer(AController* NewPlayer)
{
    ResetAllCombatPuzzles();   // Reset all combat arenas on death

    // Clean up old pawn and respawn
    if (APawn* OldPawn = PC->GetPawn())
    {
        PC->UnPossess();
        OldPawn->Destroy();
    }

    Super::RestartPlayer(NewPlayer);
}

What this does

On every player respawn (including after death), the GameMode automatically resets all CombatPuzzleSystems across the level. This ensures that if the player died during a combat arena, the walls and enemies reset properly for the next attempt.

Key Advantages

• Fully integrated into the UniversalTrigger — designers just tick a checkbox and assign a spawn point
• Consistent player rotation on respawn (forced yaw of 45°)
• Automatic reset of all combat puzzles on death
• Clean separation: Trigger handles saving • GameMode handles spawning
• Supports both normal level starts and checkpoint respawns

The Bush Puzzle is an interactive environmental puzzle where the player must use their Fire elemental ability to ignite special bushes. Once ignited, bushes burn with realistic dissolve effects, spread fire to nearby bushes, and eventually destroy themselves — opening new paths for the player and AI.

Player Hitting only one bush with fire , the rest nearby will burn too

BushPuzzle.cpp — Core Ignition Logic

BushPuzzle.cppC++
void ABushPuzzle::IgniteBush()
{
    if (bIsIgnited) return;
    bIsIgnited = true;

    // Swap to dynamic dissolve material with fire emissive
    for (int32 i = 0; i < BushMesh->GetNumMaterials(); i++)
    {
        UMaterialInstanceDynamic* DynMat = UMaterialInstanceDynamic::Create(DissolveMaterial, this);
        BushMesh->SetMaterial(i, DynMat);
        DynMat->SetVectorParameterValue("EmissiveColor", FLinearColor(1.0f, 0.3f, 0.0f, 1.0f));
        DynMat->SetScalarParameterValue("Disolve", -0.4f);
    }

    PlaySound(IgniteSound);
    SpawnNiagaraEffect(BurningEffect);

    // Start dissolve animation + destruction timer
    GetWorldTimerManager().SetTimer(DissolveTimerHandle, this, &ABushPuzzle::UpdateDissolve, 0.05f, true);
    GetWorldTimerManager().SetTimer(DestructionTimerHandle, this, &ABushPuzzle::DestroyBush, DestructionDelay, false);

    if (bCanSpreadFire)
        GetWorldTimerManager().SetTimer(FireSpreadTimerHandle, this, &ABushPuzzle::SpreadFireToNearbyBushes, FireSpreadDelay, false);
}

What this does

When the player hits the bush with a Fire ability (via TriggerIgnite()), the bush switches to a dynamic material, starts burning with Niagara particles, plays sound, and begins a timed dissolve effect. After a delay, the bush is fully destroyed.

Fire Spread System

BushPuzzle.cppC++
void ABushPuzzle::SpreadFireToNearbyBushes()
{
    TArray<FOverlapResult> OverlapResults;
    GetWorld()->OverlapMultiByObjectType(
        OverlapResults,
        GetActorLocation(),
        FQuat::Identity,
        FCollisionObjectQueryParams(ECC_WorldDynamic),
        FCollisionShape::MakeSphere(FireSpreadRadius)
    );

    for (const FOverlapResult& Result : OverlapResults)
    {
        if (ABushPuzzle* Nearby = Cast<ABushPuzzle>(Result.GetActor()))
        {
            if (!Nearby->bIsIgnited)
                Nearby->TriggerIgnite();
        }
    }
}

What this does

After igniting, the bush automatically spreads fire to all other ABushPuzzle actors within a configurable radius using a sphere overlap query. This creates satisfying chain-reaction moments where one well-placed Fire ability can clear an entire area of bushes.

Visual & Navigation Polish

• Dynamic Dissolve Material: Real-time lerp from -0.4 to 0.7 on the "Disolve" parameter for smooth burning effect
• Player Proximity Fresnel: Bushes glow with a special material when the player gets close (then revert when player leaves)
• NavMesh Integration: Collision box is set to affect navigation. Once destroyed, collision is disabled so enemies and the player can path through the cleared area.
• Niagara Effects: Burning fire + destruction burst for high visual quality

Key Features & Design Highlights

• Fully driven by Gameplay Ability System (GAS) — the Fire ability sends a tag/event to trigger ignition
• Chain-reaction fire spread creates emergent gameplay
• Smooth timed dissolve with sound + VFX
• Proper NavMesh updating so AI doesn't get stuck after bushes burn
• Proximity-based visual feedback for better player readability
• Designer-friendly: adjustable spread radius, burn delay, destruction timing, and materials all exposed in Blueprint

{

}

What this does

If the timer runs out before the puzzle is solved, the bell deactivates and notifies the PuzzleManager. This forces the player to hit all bells again within the time limit — creating a real sense of pressure and urgency.

Visual Feedback System

• Dynamic Materials: Bells switch between default, activated, and Fresnel-highlight versions.
• Player Proximity: When the player gets close, the bell and stand use glowing Fresnel materials for better readability.
• Separation of Concerns: Collision on the pole, visual bell on top, and a dedicated DetectionBox for the Water ability.

How it connects to the Puzzle Manager

Every time a bell is successfully hit or expires, it calls back into the PuzzleManager. The manager tracks all time puzzles in the level and only declares the overall puzzle solved when every bell is active at the same time (i.e. no timers have expired yet).

• Clean GAS integration instead of messy direct collision checks
• Smart material switching with proximity awareness
• Fully resettable for failed attempts
• Designer-friendly: adjustable TimeLimit, DetectionBox size, and materials exposed in Blueprint

{

    bActivated = true;

What this does

When hit correctly, the switch lights up with a persistent fire Niagara effect and notifies the PuzzleManager. The manager then checks whether this switch was the next expected one in the sequence.

Smart Visual Feedback

• Niagara Fire Effect: Activates only when the switch is correctly hit in sequence.
• Player Proximity Material: Switches to a glowing material when the player gets close.
• Dynamic Outline: In Tick(), the switch renders with custom depth (outline) when the player is nearby for better readability.

The SequencePuzzle doesn’t decide correctness itself. It simply calls OnSequencePuzzleHit(this) every time it’s activated. The PuzzleManager tracks the order using each object’s SequenceIndex and decides whether to advance, reset the whole sequence, or complete the puzzle.

Design Highlights

• Clean GAS-based interaction (consistent with other elemental puzzles)
• Persistent visual state (fire stays on for correct hits)
• Proximity-based feedback (material + outline) improves player readability
• Fully resettable — wrong order immediately turns off all fire effects
• Tick-based outline is lightweight and only checks distance to player
