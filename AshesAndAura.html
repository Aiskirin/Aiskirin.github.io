A clean and maintainable Finite State Machine (FSM) that manages the player's different behaviors (Running, Dashing, Hurt, Dead, etc.). This architecture keeps the PlayerController clean and makes it very easy to add new states in the future.

StateMachine.cs

StateMachine.cs C# 
using UnityEngine;

public class StateMachine
{
    private StateBase currentState = null;

    public void SetState(StateBase newState)
    {
        if (newState == null)
        {
            Debug.LogWarning("SetState called with null newState");
            return;
        }

            if (currentState != null)
        {
            currentState.Exit();
        }

        currentState = newState;
        currentState.Enter();
    }

    public void Tick(float deltaTime)
    {
        if (currentState == null)
        {
            Debug.LogWarning("Tick called with null currentState. Call SetState first.");
            return;
        }

        currentState.Update(deltaTime);

        if (currentState.TryFindNextState(out StateBase nextState))
        {
            SetState(nextState);
        }
    }

    public StateBase GetCurrentState() => currentState;
}

What this does

This StateMachine is the brain of the player. It follows the classic Enter → Update → Exit pattern and supports automatic transitions:

  • Clean transitions: Automatically calls Exit() on the old state and Enter() on the new one.
  • Automatic switching: Every frame, the current state can request to change to a new state via TryFindNextState().
  • Safe & robust: Includes null checks and helpful debug warnings.
  • Highly extensible: Adding a new state (e.g. Climbing, Sliding, Power-up) only requires creating a new StateBase subclass.

How to use

In your PlayerController.cs:

private StateMachine stateMachine;

void Start()
{
    stateMachine = new StateMachine();
    stateMachine.SetState(new PlayerRunState(this));
}

void Update()
{
    stateMachine.Tick(Time.deltaTime);
}

Each state (e.g. PlayerDashState, PlayerHurtState) inherits from StateBase and implements Enter(), Update(), Exit(), and TryFindNextState().

A lightweight and elegant way to define when a state should automatically transition to another state. Instead of writing messy if-else chains inside every state's Update(), each state holds a list of TransitionPairs that are checked every frame.

TransitionPair.cs

TransitionPair.cs C# 
public class TransitionPair
{
                                    public Func<float, bool> predicate;
                                    public StateBase nextState;

                                    public TransitionPair(StateBase nextState, Func<float, bool> predicate)
    {
                                    this.nextState = nextState;
                                    this.predicate = predicate;
    }
}

What this does

The TransitionPair is a simple but powerful data structure that pairs a condition with a target state.

  • predicate: A function that receives deltaTime and returns true when the transition should happen.
  • nextState: The state to switch to when the predicate becomes true.

This keeps transition logic clean, reusable, and easy to read — no more giant if-else chains in your states.

    public override void Enter() { ... }

    public override void Update(float deltaTime)
    {
      // Example transitions
      if (Input.GetKeyDown(KeyCode.Space))
      stateMachine.SetState(new PlayerDashState(player));

      // Or using TransitionPair (recommended)
    }

    public override bool TryFindNextState(out StateBase next)
    {
        next = null;

        foreach (var transition in transitions)
        {
             if (transition.predicate(deltaTime))
            {
                next = transition.nextState;
                 return true;
            }
        }
        return false;
    }
}

You can even pre-populate the transitions list in the state's constructor for a fully data-driven approach.

Why it's great

  • Keeps each state focused only on its own behavior
  • Makes transitions declarative and easy to debug
  • Very easy to add new transitions without touching core state logic
  • Works beautifully with the StateMachine.Tick() system you already have

The foundation of the entire state machine system. StateBase provides a clean, reusable structure for all player states while automatically handling transitions via TransitionPair and tracking how long the player has been in the current state.

StateBase.cs

StateBase.cs C# 
public abstract class StateBase
{
    protected readonly List<TransitionPair> Transitions = new();

    private float timeInState = 0f;
    public float TimeInState => timeInState;

    public void AddTransition(StateBase to, Func<float, bool> predicate)
    {
        Transitions.Add(new TransitionPair(to, predicate));
    }

    public bool TryFindNextState(out StateBase nextState)
    {
        foreach (var t in Transitions)
        {
            if (t.predicate(timeInState))
            {
                nextState = t.nextState;
                return true;
            }
        }
        nextState = null;
        return false;
    }

    protected virtual void OnEnter() { }
    protected virtual void OnExit() { }
    protected virtual void OnUpdate(float deltaTime) { }
}

What this does

StateBase is the heart of your clean FSM architecture. It gives every state:

  • Built-in transition system using TransitionPair
  • Time tracking — TimeInState tells you how long you've been in this state (very useful for timed behaviors like invincibility frames, attack cooldowns, etc.)
  • Clean lifecycle methods: OnEnter(), OnExit(), and OnUpdate()
  • Easy transition registration via AddTransition()

How states inherit from it

public class PlayerRunState : StateBase
{
    private readonly PlayerController player;

    public PlayerRunState(PlayerController player)
    {
        this.player = player;

        // Register transitions
        AddTransition(new PlayerDashState(player), dt => Input.GetKeyDown(KeyCode.Space));
        AddTransition(new PlayerHurtState(player), dt => player.IsHit);
    }

    protected override void OnEnter()
    {
        player.Animator.SetBool("IsRunning", true);
    }

    protected override void OnUpdate(float deltaTime)
    {
        player.HandleMovement();
    }

    protected override void OnExit()
    {
        player.Animator.SetBool("IsRunning", false);
    }
}

Why it's great

  • Eliminates duplicate transition code across states
  • TimeInState makes timed actions (dash cooldown, hurt recovery, etc.) trivial
  • Transitions are declarative and easy to read
  • Keeps every state focused, clean, and maintainable
  • Works perfectly with your existing StateMachine class
